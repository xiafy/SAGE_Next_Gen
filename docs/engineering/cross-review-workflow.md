## 核心原则

**设计者不审自己的设计，编码者不审自己的代码。**

来源：
- SAGE 四方审计 2.2/5 的根因——自我审查无效（RETRO_2026-03-03.md）
- OpenAI Harness Engineering——"Enforce invariants, not implementations"
- Mr. Xia 指示：Opus 主 session 只做协调者，所有执行角色独立 sub-agent

## 四步交叉流

```
┌─────────────────────────────────────────────────┐
│                  协调者（主 Opus）                │
│  职责：准备上下文、调度 sub-agent、传递产出、     │
│        决策回退、向夏总汇报                       │
└────────┬──────────┬──────────┬──────────┬────────┘
         │          │          │          │
    Step 1      Step 2     Step 3     Step 4
   Opus-A      Codex-A    Codex-B    Opus-B
   设计方案    审查设计    编码实现    审查代码
         │          │          │          │
         ▼          ▼          ▼          ▼
      方案文档   审查报告    代码+测试   审查报告
                    │                     │
              通过/不通过            通过/不通过
              不通过→Step1           不通过→Step3
                                   (架构问题→Step1)
```

### Step 1: Opus-A（设计者）

- **模型**: Opus
- **输入（由协调者准备，⭐最关键步骤）**:
  - 任务目标（做什么、为什么做）
  - 项目现状（相关源码片段、目录结构、现有测试）
  - 约束条件（技术栈、依赖、不可改动的部分）
  - 历史上下文（相关 DEC 决策、过去的教训）
  - 参考资料（如 OpenAI Harness Engineering 相关原则）
- **输出**:
  - 设计方案文档（含接口定义、边界条件、取舍权衡）
  - 验收标准（怎么算"做完了"）
- **注意**: 协调者准备上下文的质量决定整条链路成败。上下文不足或方向偏 → 后续全部失效。

### Step 2: Codex-A（设计审查）

- **模型**: Codex
- **输入**:
  - Step 1 产出的设计方案
  - 相关源码（与设计方案涉及的文件）
  - 审查清单（见下方"设计审查清单"）
- **输出**:
  - 审查报告（通过/不通过 + 具体问题列表）
  - 如不通过：修改建议（具体到"哪里改、改成什么"）
- **审查视角**: 实现可行性、技术约束、成本合理性、遗漏场景

### Step 3: Codex-B（实现者）

- **模型**: Codex
- **输入**:
  - 定稿方案（经 Step 2 审查通过的版本）
  - 项目完整访问（workdir 设为项目根）
  - 必读文件清单（协调者指定）
- **输出**:
  - 代码实现 + 对应测试
  - 实现说明（做了什么、为什么这样做、已知限制）
- **约束**: 只实现方案中定义的内容，不自行扩展范围

### Step 4: Opus-B（代码审查）

- **模型**: Opus
- **输入**:
  - 定稿方案（作为意图基准）
  - Step 3 的代码 diff（`git diff`）
  - 测试运行结果
  - 审查清单（见下方"代码审查清单"）
- **输出**:
  - 审查报告（通过/不通过 + 具体问题列表）
  - 如不通过：问题分级（小问题→Codex-B 修 / 架构问题→退回 Step 1）

## 风险分级

| 级别 | 判定标准 | 流程 |
|------|---------|------|
| **高** | 重构核心逻辑、新增门控机制、改状态机、改架构 | 完整四步 |
| **中** | 新增测试、文档迁移、配置文件、非核心脚本 | 三步（Step 1→3→4，跳过设计审查） |
| **低** | 文档修正、gitignore、版本号、注释 | 协调者直接执行 |

## 审查清单

### 设计审查清单（Step 2 Codex-A 使用）

```
□ 方案在当前技术栈下可实现（不依赖不存在的工具/库）
□ 边界条件已覆盖（空输入、异常路径、并发场景）
□ 与现有代码/架构不冲突（不破坏已有功能）
□ 成本合理（不过度工程化，不为 1% 场景增加 50% 复杂度）
□ 验收标准明确且可机械化验证
□ 遗漏场景检查（设计者可能忽略的 edge case）
```

### 代码审查清单（Step 4 Opus-B 使用）

```
□ 代码实现与设计方案意图一致（不多不少）
□ 边界条件在代码中有对应处理
□ 测试覆盖了方案中定义的验收标准
□ 无架构退化（不引入新的技术债）
□ 命名/结构符合项目约定
□ 错误信息可读且可操作（特别是给 Agent 看的 linter 报错）
```

## 回退机制

| 审查结果 | 动作 |
|---------|------|
| Step 2 通过 | → Step 3 |
| Step 2 不通过（小问题） | 协调者修正方案 → 重新提交 Step 2 |
| Step 2 不通过（方向问题） | 协调者重新准备上下文 → 重走 Step 1 |
| Step 4 通过 | → Commit + Push |
| Step 4 不通过（小问题） | 列出修改点 → Codex-B 修正 → 重新 Step 4 |
| Step 4 不通过（架构问题） | 退回 Step 1 重新设计 |

---

## 协调者职责

协调者（主 Opus session）**不参与设计和编码**，职责是：

1. **准备上下文包**: 为每个 sub-agent 准备精确的输入（这是最核心的工作）
2. **调度 sub-agent**: 按顺序 spawn，传递前序产出
3. **传递和过滤**: 在步骤之间传递产出，过滤无关信息
4. **决策回退**: 审查不通过时判断回退到哪一步
5. **向夏总汇报**: 每步完成后简要汇报结果
6. **记录决策**: 过程中的关键决策记入 DECISIONS.md

## 并行策略

- 同一 Phase 内：四步串行（有依赖）
- 不同模块/文件：可并行走各自的四步
- 例：Phase 1 的 pre-commit.sh 和 deploy.sh 可并行两条线
