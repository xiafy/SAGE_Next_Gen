# Spec-Driven + Test-Driven Workflow

> 目标：先定义「是什么」，再定义「怎么验」，最后写「怎么实现」。

---

## 核心理念

**Spec（规格） → Test（测试） → Code（实现）**

先写清行为边界和验收标准，再通过测试锁定预期，最后实现代码。

---

## 基本原则

1. **规格先行**：每个功能开发前先产出规格文档，存放在 `specs/`。
2. **测试护航**：规格确定后先写测试（或测试骨架），再写实现。
3. **质量为纲**：粒度可灵活，但交付质量和行为正确性不可妥协。

---

## 规格层次

| 层次 | 规格形式 | 测试形式 |
|------|---------|---------|
| 产品层 | PRD / User Story + 验收标准 | E2E 场景测试 |
| API 层 | 接口契约 / OpenAPI | 契约测试 + 集成测试 |
| 模块层 | 函数签名 + 行为描述 | 单元测试 |

---

## AI 辅助策略

- 允许基于 spec 自动生成测试骨架。
- 允许基于 spec + test 直接辅助实现代码。
- AI 生成内容必须回到规格与测试验收，不可跳过验证。

---

## 标准工作流（6 步）

1. **需求对齐**：明确做什么、为谁做、业务价值是什么。
2. **撰写 Spec**：写入 `specs/`，定义输入/输出/边界/验收标准。
3. **编写测试**：基于 spec 先写测试或最小测试骨架。
4. **实现代码**：以“让测试通过”为目标完成实现。
5. **验证交付**：测试全绿 + 人工验收主路径与异常路径。
6. **复盘沉淀**：更新文档、决策和可复用最佳实践。

---

## 执行要求

- 不在未读规格的情况下直接改代码。
- 需求变更先更新 spec，再更新 test 与 code。
- 合并前必须证明：规格可追溯、测试可复现、实现可解释。
