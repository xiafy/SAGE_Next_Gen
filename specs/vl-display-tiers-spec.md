# SAGE 菜品信息分层展示规格

**版本**: v1.0  
**日期**: 2026-03-02  
**状态**: 待 Mr. Xia 确认  
**背景**: 两阶段 SSE 优化的前置产品设计，确定"哪些信息必须首屏展示、哪些可以延迟或按需加载"，以及质量标准。

---

## 一、当前 MenuItem 字段全景

```
├── 核心识别（VL 直接输出）
│   ├── nameOriginal      原文菜名
│   ├── nameTranslated    翻译菜名（中/英）
│   ├── priceText         价格原文（如 "70.-"）
│   ├── price             价格数值（解析自 priceText）
│   └── category          所属分类原文
│
├── 菜单元数据（VL 直接输出）
│   ├── menuType          restaurant/bar/dessert/...
│   ├── detectedLanguage  菜单语言
│   ├── currency          货币单位
│   └── priceLevel        1/2/3
│
└── 语义字段（Enrich 补全）
    ├── brief             一句话简介（≤20字）
    ├── briefDetail       展开详情（≤40字，用户点击才展开）
    ├── spiceLevel        辣度 0-5
    ├── allergens         过敏原数组 [{type, uncertain}]
    ├── dietaryFlags      饮食标签 [halal/vegetarian/vegan/...]
    └── calories          卡路里（当前未实现，null）
```

## 二、信息分层展示设计

### Tier 1 — 首屏必须展示（VL 完成后立即渲染，~8s）

| 字段 | 展示位置 | 理由 |
|------|---------|------|
| `nameTranslated` | 卡片主标题 | **必须**：用户核心需求，不知道菜名就无法做任何决策 |
| `nameOriginal` | 副标题（小字） | **必须**：展示给服务员用，原文才有意义 |
| `priceText` | 右上角 | **必须**：价格是点单决策的第一因素 |
| `category` | 分类导航 | **必须**：分组浏览的前提 |
| `currency` + `priceLevel` | 菜单顶部提示 | **推荐**：帮助用户建立价格预期 |

**Tier 1 数据来源**：VL 单次调用即可满足，无需 Enrich。

**Tier 1 质量底线**（必须满足才能展示）：
- `nameTranslated` 非空
- `priceText` 如存在则必须与图片一致，不能编造
- 识别失败（items 为空）→ 不展示 Tier 1，直接进入重试流程

---

### Tier 2 — 自动补全（Enrich 完成后静默更新，~+10s）

| 字段 | 展示位置 | 理由 |
|------|---------|------|
| `brief` | 菜名下方小字（line-clamp-1） | 帮助用户快速判断"这是什么"，增强扫描效率 |
| `spiceLevel` | 标签行（🌶 图标） | 辣度是高频决策因子，不需要用户主动请求 |
| `allergens` | 标签行 + 顶部警告条 | ⚠️ **安全关键**，见下方专项说明 |
| `dietaryFlags` | 标签行（绿色 halal/素食） | 饮食限制用户的核心需求 |

**allergens 的特殊处理**：

过敏原是 SAGE 的安全关键字段，需要特殊对待：
- **有偏好设置的用户**：Tier 2 加载完成前，匹配过敏原的卡片显示"正在检查过敏原…"状态，而非静默空白
- **无偏好设置的用户**：Tier 2 加载完成后静默填充即可
- **Enrich 失败时**：不能显示"无过敏原"，必须显示"过敏原信息不可用"

---

### Tier 3 — 按需加载（用户主动点击"详情"）

| 字段 | 触发方式 | 理由 |
|------|---------|------|
| `briefDetail` | 点击卡片"详情▶"按钮 | 40字详情是深度兴趣，非刚需，不用预加载 |
| `calories` | （当前 null，未来功能） | 需要专门的营养数据库或模型，不在 Enrich 范围 |
| 菜品图片 | 点击卡片（未来） | 需要独立图片搜索，与识别链路分开 |
| AI 深度解读 | 点击"问 AI" | 转入 Chat，属于 F06 探索型意图 |

## 三、质量标准（与速度同等重要）

> **Mr. Xia 原则**：速度和质量缺一不可。快但错的信息比慢但准的更危险。

### 3.1 各字段质量要求

| 字段 | 质量等级 | 错误容忍度 | 说明 |
|------|---------|-----------|------|
| `nameTranslated` | 🔴 关键 | 零容忍 | 翻译错误直接导致用户点错菜 |
| `priceText` | 🔴 关键 | 零容忍 | 价格必须与菜单图片完全一致，不得编造 |
| `allergens` | 🔴 安全关键 | 漏报 > 误报 | 宁可多标"可能含有"，绝不漏报已知过敏原 |
| `spiceLevel` | 🟡 重要 | 允许 ±1 级误差 | 影响体验，不影响安全 |
| `dietaryFlags` | 🟡 重要 | halal/vegan 漏报不可接受 | 宗教/伦理敏感 |
| `brief` | 🟢 增强 | 允许不够准确 | 辅助信息，错了也不危险 |
| `briefDetail` | 🟢 增强 | 宽松 | 用户主动请求，期望值较低 |

### 3.2 当前已知质量问题（本次测试发现）

1. **变体菜品 brief 退化**：Pad Thai 系列中，"猪肉"变体的 brief 输出"猪肉+泰式炒河粉"而非"泰式炒河粉猪肉版，酸甜微辣"。根因：Gemini 对变体菜品识别结构不稳定。
2. **categories 数组为空**：两阶段和单次测试中，Gemini 都倾向于返回空的 `categories` 数组，分类信息全部塌缩为"其他"。Worker 的 `normalizeLooseResult()` 会从 items 的 category 字段重建，但 nameTranslated 全为原文，未翻译。
3. **价格准确性未系统测试**：目前仅目测验证，需要 ground truth 对比测试。
4. **allergens 准确性未测试**：这是最高风险的未知项——当前测试仅验证了"功能存在"，没有验证"是否正确"。

### 3.3 质量测试计划（优化实施前必须完成）

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 价格准确率 | 用 5 张菜单图人工标注 ground truth，对比 VL 输出 | ≥95% 价格完全匹配 |
| 翻译准确率 | 同上，母语用户评估翻译 | ≥90% 翻译可接受 |
| allergens 召回率 | 用含明确过敏原的菜单测试（如"含花生"字样） | 召回率 ≥ 80%，漏报率 < 10% |
| allergens 误报率 | 用不含过敏原的菜单测试 | 误报率 < 20% |
| spiceLevel 准确率 | 用已知辣度等级的泰/川/印度菜单 | 与人工评估偏差 ≤1 级 |
| categories 翻译 | 检查 normalizeLooseResult 重建的分类翻译 | 分类名翻译正确率 ≥ 90% |

## 四、两阶段 SSE 的实施前提

基于以上分析，两阶段 SSE 方案可以实施，但有以下**前提条件**：

1. ✅ Tier 1（VL 输出）质量测试通过后才能 partial_result 推送——不能把低质量的 Tier 1 内容快速推给用户
2. ⚠️ Tier 2 中的 allergens 需要在 Enrich 完成时才推送，不能在 Tier 1 里以"空数组"展示（会被误读为"无过敏原"）
3. ❓ 质量测试结果影响是否需要在 prompt 层面优化再实施 SSE

## 五、产品决策记录（Mr. Xia 确认，2026-03-02）

### D1｜Tier 1/2 拆分策略
**决策**：不轻易下结论，通过 A/B Test 验证。  
- A 组：等待完整结果（~19s）一次性展示  
- B 组：8s 展示 Tier 1，+10s 静默补全 Tier 2  
- 需测试用户是否真正更偏好 B 组（不完整信息可能带来更大的焦虑感）

### D2｜allergens 加载状态
**决策**：有过敏原偏好的用户，Enrich 完成前卡片上显示「正在检查过敏原…」。  
具体形态（文字 / 图标 / 动画）后续 UI 评审时确定。  
**底线**：绝不能以"空白"暗示"无过敏原"。

### D3｜categories 缺失处理
**决策**：分两种情况：  
- **全部缺失**（整张菜单无分类）→ 不展示分类区域，直接展示菜品列表  
- **部分缺失**（部分菜品有分类、部分没有）→ 两个备选方案，待进一步评估：  
  - 方案 a：无法识别分类的菜品归入「其他」  
  - 方案 b：调用 AI 根据菜品名称推理出合理分类  
**禁止**：直接显示未翻译的原文分类名（如 "Main Dishes Thai Cuisine"）。

### D4｜变体菜品展示方式
**决策**：拆分成独立卡片，每个变体一张。  
**理由**：菜单印刷为节省空间做了合并，但用户在 SAGE 里选菜需要明确的独立条目。  
**覆盖的场景**（需在 VL Prompt 中明确处理）：  
- 蛋白质变体：猪肉 / 鸡肉 / 虾 / 素食  
- 规格变体：大份 / 小份  
- 销售方式变体：红酒杯卖 / 瓶卖  
- 类似结构均拆分，不折叠  
**影响**：VL Prompt 需要明确指示"每个独立可点单的选项作为一条 item 输出"。
