## 1. 产品概述

SAGE 是面向全球旅行者的餐饮智能体。

**核心价值主张**：在菜单超过 30% 是非母语的陌生餐厅，帮用户通过 AI 对话完成点餐决策——无论是深入了解单道菜，还是一键生成整套用餐方案。

**MVP 范围**：
- 用户群：中国旅行者 + 英语系欧美旅行者
- 形态：手机 Web App，无需下载，无需登录
- 语言：中英双语 UI（系统语言自动切换 + 支持手动切换）
- **用户路径：实现 Path A（主路径）+ Path C（补充菜单）**
- Path B（随便聊聊）移入 Backlog，MVP 不实现

## 2. 用户画像

### Persona A：阿明（中国旅行者）
- **背景**：30 岁，上海互联网从业者，每年出境 3-4 次（日韩东南亚为主）
- **场景**：在曼谷夜市，菜单全是泰文
- **现有行为**：拍照发给 ChatGPT 翻译，但要来回切换 App，且只能翻译不能推荐
- **核心诉求**：先看全量菜单有个大致了解，再让 AI 推荐
- **痛点**：翻译工具只翻译，不推荐；通用 AI 步骤多；不懂菜单规则导致尴尬
- **设备**：iPhone，微信重度用户，不愿安装新 App

### Persona B：Sarah（美国旅行者）
- **背景**：28 岁，纽约设计师，第一次去日本旅行
- **场景**：在东京居酒屋，菜单全是日文，服务员英语有限
- **现有行为**：用 Google Lens 翻译，不知道菜是什么样、适不适合自己
- **核心诉求**：了解菜品文化背景，避免踩雷（过敏、口味不合）
- **痛点**：不了解日本饮食文化；担心点到不能吃的东西
- **设备**：Android，Safari/Chrome 用户

## 3. 核心用户旅程

### 旅程 A：扫描菜单路径（主路径）`[MVP]`

```
[进入 App]
    ↓ GPS 授权请求（立即，静默）
[Home]
    ↓ 点击「扫描菜单」
[Scanner 全屏相机页面]
    ├─ 拍照（支持连续拍摄多张）
    ├─ 从相册选择
    └─ 预览 / 删除 / 确认（最多 5 张）
         ↓ 确认发送，立即跳转
[AgentChat]
    ├─ Icebreaker（AI 先问：几位？有忌口？）
    ├─ 菜单识别完成 → AI 开场介绍 + 推荐
    └─ 用户进入两种模式之一（自然涌现，不显式区分）：
         ├─ 探索模式：深入了解某道菜的味道/文化/历史
         └─ 方案模式：AI 根据约束条件输出完整用餐方案
              ↓ 确认/调整后
[点餐单] → [展示给服务员]
```

### 旅程 B：直接对话 `[Backlog — MVP 不实现]`

> 场景：在路上问"泰国菜我该注意什么"等。MVP 后规划。

### 旅程 C：补充菜单路径 `[MVP]`

```
[AgentChat 进行中]
    ↓ 点击「📷 补充菜单」
[Scanner 全屏相机页面]（同 Path A）
    ↓ 拍摄酒单 / 甜品单 / 套餐规则页
[AgentChat]
    └─ AI 自动合并菜单，提示新增内容，继续对话
```

### 旅程 D：菜单总览自主浏览路径 `[MVP]` (DEC-043 新增)

```
[AgentChat]
    ↓ 点击「📋 查看全部菜单」
[Explore 菜品探索]
    ├─ 按分类浏览所有菜品
    ├─ 逐一挑选感兴趣的菜品加入已选列表
    └─ 选完后两个出口：
         ├─ 直接展示给服务员 → [Waiter Mode]
         └─ 带已选菜品咨询 AI → [AgentChat]
              └─ AI 根据已选菜品给出搭配建议/调整方案
                   ↓ 确认后
              [点餐单] → [展示给服务员]
```

### 3.1 Chat 两种核心意图（DEC-043）

Chat 界面支持两种用户意图，由 AI 根据上下文自然涌现（Scenario-Free 原则），不在 UI 上做显式区分：

**意图 A：探索型** — "这道菜是什么？"
- 用户对某道菜感兴趣，想深入了解味道、做法、文化背景、食材
- AI 扮演美食百科，深度讲解单道菜品的价值
- 触发方式：从 Explore 点击某道菜进入、或在 Chat 中直接提问

**意图 B：方案型** — "帮我配一套菜"
- 用户给出约束条件（人数、忌口、过敏原、预算、口味偏好等）
- AI 输出**完整结构化用餐方案**，课程结构由 AI 根据餐饮文化动态生成（DEC-059）——不限于西餐的"前菜→主菜→甜品→酒水"
- 方案包含：每道菜的推荐理由 + 整体搭配逻辑
- 用户可在方案基础上微调（换掉某道、增减菜品）
- 确认后**整套方案一键加入订单**

## 4. 功能规格

---

### F01 — Home

**用户目标**：打开 App 立即看到入口，1 秒内开始操作。

**功能描述**：
- 右上角：「⚙️ 设置」图标（进入偏好管理 + 语言切换）
- 动态问候语（基于时段：早安 / 午好 / 晚好 / 深夜好）
- **无 Tab Bar**：纯任务流导航（DEC-040）
- 无注册/登录引导，打开即用
- **会话感知双状态**：
  - **无进行中会话**：主入口「📷 扫描菜单」（大按钮，视觉焦点）
  - **有进行中会话**（menuData 存在）：「🍽 继续上次用餐」→ Chat + 「🔄 新的一餐」→ 清空后进 Scanner

**验收标准**：
- AC1: 首屏可交互 < 2 秒（4G 网络）
- AC2: 主入口在 375px-430px 屏宽下清晰可见，无需滚动
- AC3: 语言根据系统语言自动显示
- AC4: 设置图标可访问，点击进入设置页
- AC5: 有进行中会话时显示「继续上次」和「新的一餐」两个入口
- AC6: 「新的一餐」清空会话数据后进入 Scanner

**优先级**：`[MVP]`

---

### F02 — Scanner（全屏相机页面）

**用户目标**：快速完成菜单拍摄，支持多张，可删除重拍，体验流畅。

**功能描述**：
- **独立全屏页面**（不是底部弹层）
- 进入后默认激活相机
- 顶部：已拍张数显示 + 关闭按钮
- 底部操作区：
  - 快门按钮（拍照）
  - 相册按钮（选择已有照片）
  - 缩略图预览条（已选图片，可点击删除）
- 支持最多 **5 张**图片
- 「确认」按钮：确认后关闭 Scanner，跳转 AgentChat 并开始上传识别
- **两个入口**：
  1. Home「扫描菜单」按钮（首次进入，Path A）
  2. AgentChat 内「📷 补充菜单」按钮（对话中途，Path C）

**验收标准**：
- AC1: 从 Home 点击到 Scanner 相机激活 < 2 秒
- AC2: 支持 JPG / PNG / HEIC 格式
- AC3: 最多 5 张，超过时禁用快门并提示
- AC4: 已选图片显示缩略图，可单张删除
- AC5: 相机权限被拒时显示引导说明，相册入口仍可用
- AC6: 图片上传前压缩至 < 2MB（不影响识别质量）
- AC7: Path C 进入时保持 AgentChat 对话历史，返回后可继续对话
- AC8: 补充菜单识别结果与已有菜单合并时，按 dishId 去重（优先级：原文菜名 + 价格作为 merge key）；重复菜品保留已有数据，不覆盖；合并后 AI 收到更新通知

**优先级**：`[MVP]`

---

### F03 — 菜单识别（AI Vision）

**用户目标**：AI 准确理解菜单内容，为对话和推荐提供基础。

**功能描述**：
- **两阶段架构**（DEC-042）：
  - **Step 1 — OCR**：qwen3-vl-flash 提取菜品名称（原文+翻译）、价格、分类，超时 20s
  - **Step 2 — 语义补全**：qwen3.5-flash（文本模型）为每道菜补充 brief、briefDetail、allergens、dietaryFlags、spiceLevel，超时 15s，best-effort（失败不阻塞）
- 多张图片合并识别，按类别整合，重复菜品去重
- 同时输出：`menuType`、`detectedLanguage`、`priceLevel`（1-3 档）、`currency`
- 识别过程中 AgentChat 不阻塞，显示 Icebreaker 对话

**识别失败处理**：
- 图片模糊 → 提示"图片有点模糊，建议重新拍一张，光线好一点更准"+ 「重新拍摄」按钮
- 语言暂不支持 → 提示"这个语言我暂时识别不好，建议换张更清晰的图或换个角度"
- 网络/API 异常 → 提示"网络好像有点问题，稍后再试试"+ 「重试」按钮
- **不引导用户手动输入菜单**（输入法问题 + 失去使用 SAGE 的意义）

**验收标准**：
- AC1: 支持语言：中 / 英 / 日 / 韩 / 泰 / 法 / 西班牙 / 意大利 / 德语
- AC2: 标准印刷菜单识别成功率 ≥ 90%
- AC3: 识别时间 < 10 秒（P90，正常网络）
- AC4: 所有 AI 返回数据经 zod schema 校验，校验失败时有降级处理
- AC5: 失败提示引导用户重新拍摄，不出现技术错误信息

**优先级**：`[MVP]`

### F04 — 4+1 维感知数据采集

**用户目标**：AI 理解我当下的完整场景，不只是菜单内容。

#### F04-A 时间感知 `[MVP]`
- 自动读取系统时间，注入 AI 上下文
- 时段映射：07-10 早餐 / 11-14 午餐 / 17-21 晚餐 / 21+ 夜宵

#### F04-B 空间感知 `[MVP]`
- 进入 App 时立即请求 GPS 授权
- 权限被拒：**静默跳过，不显示任何提示**，AI 推荐不考虑位置维度
- 权限通过：获取大致区域（城市级），注入 AI 上下文

#### F04-C 天气感知 `[Sprint 2]`
- 依赖 GPS，Sprint 2 实现

#### F04-D 历史记忆 `[MVP]`
- localStorage 持久化用户偏好
- 下次打开时注入 AI 上下文，AI 自然融入，不机械念清单

**验收标准**：
- AC1: 所有感知数据采集失败时，主流程不中断
- AC2: GPS 被拒后无任何弹窗或提示
- AC3: 历史记忆跨 session 持久（同浏览器/设备）

**优先级**：`[MVP]`

### F06 — AgentChat 对话界面

**用户目标**：通过自然对话完成点餐决策——深入了解菜品，或让 AI 生成完整用餐方案。

**功能描述**：

**Pre-Chat 机制（DEC-027，升级自 Icebreaker）**：
- 图片确认后立即跳转 AgentChat，不等识别完成
- AI 立即发出第一条消息（本地生成，毫秒出现），随即进入 **Pre-Chat 模式**
- Pre-Chat 模式：AI（Qwen3.5-Flash）**主动多轮引导**用户说出需求，而非静态等待
  - 每轮只问一件事：用餐人数→忌口→口味偏好→今日心情…
  - 对话自然、像朋友问候，不像问卷填写
  - AI 随时从对话中提炼偏好（inline preferenceUpdates）
- 识别完成时（HANDING_OFF）：
  - 将全量 Pre-Chat 对话历史 + **结构化偏好摘要** + 菜单数据，一并喂给主 Chat AI（Qwen3.5-Plus）
  - 结构化偏好摘要必传字段：`{diners: number, restrictions: string[], allergies: string[], tastes: string[], mood: string}`，确保主 Chat 不依赖对 Pre-Chat 原文的"理解"
  - 主 Chat AI **不重复问已回答的问题**，直接基于完整信息给出推荐
  - 用户感知：AI 第一条正式消息就精准切中需求
  - **竞态处理**：若识别在 Pre-Chat 某轮对话中途完成，等待当前轮次结束（用户发送 + AI 回复）后再 handoff，不打断用户输入

**两种核心意图（DEC-043）**：
Chat 界面支持两种意图，由 AI 根据上下文自然涌现，不做 UI 显式区分：

- **探索型**：用户询问某道菜的味道、做法、文化背景、食材等。AI 作为美食百科深度讲解。可从 Explore 点击某菜品触发，也可在 Chat 中直接提问。
- **方案型**：用户给出约束条件（人数、忌口、过敏、预算等），AI 输出**完整结构化用餐方案**：
  - 方案包含完整用餐流程，课程结构由 AI 根据当前餐饮文化动态生成（DEC-059），不硬编码西餐顺序
  - 每道菜附推荐理由，整体附搭配逻辑说明
  - 使用**方案卡片（MealPlanCard）**组件呈现（通过流式输出末尾的 JSON 代码块提取），支持：
    - 逐道菜查看详情
    - 替换某道菜：点击「🔄 换一道」→ 自动发消息给 AI → AI 回复包含替代品的新 MealPlanCard（AI 驱动，非 UI 列表选择）
    - 增删菜品
    - 「整套加入订单」一键操作：将 MealPlanCard 中的全部菜品写入 Order（F08），MealPlanCard 是 AI 的"提案"，Order 是用户的"决定"（DEC-055）

**方案型输出格式**（DEC-052 v2）：
- AI 回复采用**流式输出**：自然语言叙事（搭配逻辑、推荐理由）逐字流式打出，末尾输出 JSON 代码块
- 前端在流式中检测到 ` ```json ` 时显示"🍽 正在生成方案…"占位
- 流式结束后提取最后一个 JSON 代码块，解析并渲染 MealPlanCard
- 分级 fallback：L1 完整解析→MealPlanCard / L2 jsonrepair→简化卡片 / L3 无法解析→纯文字+「重新生成方案」按钮
- MealPlanCard JSON schema 定义见 `shared/types.ts` 的 `MealPlan` 类型

**AI 直接操作 Order（DEC-058）**：
- AI 在对话中可直接操作用户的 Order（添加/删除/修改数量），无需用户手动到 F08 操作
- 触发方式：用户在 Chat 中说"把这道加上"/"去掉那个汤"/"主菜换成牛排"等
- AI 操作后在对话中确认变更内容（如"已帮你加入 Tom Yum Goong x1"）
- Order 变更实时同步到 F08 点餐单

**对话界面**：
- 消息气泡（AI / 用户，区分视觉）
- AI 快捷回复按钮（每次回复后动态生成 2-4 个，非固定）
- 菜品推荐**必须使用 DishCard 组件**内嵌在对话流中（不用纯文字描述菜品）
- 方案推荐**使用 MealPlanCard 组件**（整套菜品 + 搭配逻辑 + 一键加入）
- 推荐卡片（DishCard）：翻译菜名 + 原文菜名 + 概要 + 价格 + 过敏原/标签 + 「➕ 加入」按钮
- AI 的推荐理由以卡片下方小字形式展示，不作为独立聊天气泡
- 底部输入栏含「📷 补充菜单」按钮（Path C 入口）
- 「📋 查看全部菜单」入口（进入 F07 探索视图）
- 右上角点单 badge（有菜品时显示数量）→ 进入 F08 点餐单
- ← 返回 Home 需二次确认（"退出会清空当前会话，确定吗？"）（DEC-040）

**验收标准**：
- AC1: Pre-Chat 第一条消息在进入 AgentChat 后 < 1 秒出现（本地生成）
- AC2: Pre-Chat 阶段 AI 响应 < 1.5 秒（Qwen3.5-Flash）
- AC3: 菜单识别完成后主 Chat 首条消息 < 5 秒（P90）
- AC4: 主 Chat 首条消息不重复询问 Pre-Chat 已回答的问题
- AC5: 快捷按钮由 AI 根据上下文动态生成
- AC6: 切换到探索视图再返回，对话历史完整保留
- AC7: 网络中断时，已有对话不丢失
- AC8: 方案型回复使用 MealPlanCard，课程结构由 AI 根据餐饮文化动态生成（DEC-059），不硬编码西餐顺序
- AC9: MealPlanCard 支持逐道替换（AI 驱动：点击替换 → AI 回复新方案）和「整套加入订单」
- AC10: AI 自动识别用户意图（探索/方案），无需用户显式选择。意图切换信号：出现人数/预算/忌口 ≥2 个约束时优先方案型；单菜提问回到探索型。AI 在同一会话中可自然切换
- AC11: 方案型回复流式输出，叙事文字逐字可见，MealPlanCard 在流式结束后从 JSON 代码块提取渲染
- AC12: 流式中检测到 JSON 代码块开头时显示"正在生成方案"占位；分级 fallback（L1 卡片/L2 简化/L3 纯文字+重试）
- AC13: AI 可在对话中直接操作 Order（增/删/改数量），操作后在对话中显示变更确认（DEC-058）

**优先级**：`[MVP]`

### F07 — 菜品探索视图（全量菜单）

**用户目标**：自主浏览完整菜单，逐一挑选菜品，选完后可直接展示或咨询 AI。

**背景**：部分用户习惯先看全量菜单再做决策（符合 Mr. Xia 本人习惯）。该功能可提升初次使用信任度，同时解决语言障碍。

**功能描述**：
- 从 AgentChat「📋 查看全部菜单」进入
- **"全部"tab 按分类分组显示**（每组有分类标题），非"全部"tab 平铺显示
- 每道菜：原文菜名 + 翻译 + 价格 + brief 概要
- 点击展开详情（briefDetail 文化背景 / 过敏原 / 辣度 / 卡路里）
- 每道菜有「➕ 加入」按钮，已加入的菜品显示 +/- 数量控件
- 不被任何分类引用的菜品归入"其他"分类
- **底部操作栏**（有已选菜品时出现）：
  - 显示已选数量和估算总价
  - 「展示给服务员」→ 直接进入 Waiter Mode（DEC-043）
  - 「咨询 AI」→ 携带已选菜品上下文返回 AgentChat，AI 自动给出搭配建议/调整方案（DEC-043）
- 「← 返回对话」按钮随时可返回 AgentChat
- **空状态**：若菜单识别结果为空或菜品数 = 0，显示引导页："暂未识别到菜品，试试重新拍一张？"+ 「📷 重新拍摄」按钮

**验收标准**：
- AC1: 探索视图内操作不丢失 AgentChat 对话历史
- AC2: 菜品分类由 AI 识别决定，不硬编码
- AC3: 返回 AgentChat 时定位到最新消息
- AC4: "全部"tab 下菜品按分类分组，每组有标题
- AC5: 点餐单 badge 显示总数量（非种类数）；实现：`orderItems.reduce((sum, i) => sum + i.quantity, 0)`
- AC6: 已选菜品 > 0 时底部操作栏自动出现（DEC-043）
- AC7: 「咨询 AI」携带已选菜品进入 Chat：前端插入 SelectedDishesCard（系统消息样式）+ 结构化 system message（含 newlySelected + existingOrder）。AI 首条回复为事实摘要（数量/分类/预估总价）+ 开放式引导，区分"新选"和"已有"，不主动分析搭配（DEC-053 v2）。**空选择边界**：若用户未选任何菜品，不插入 SelectedDishesCard，直接返回 Chat 正常对话
- AC8: 「展示给服务员」从 Explore 直接进入 Waiter Mode（DEC-043）
- AC9: SelectedDishesCard 使用系统消息样式（非用户气泡/非 AI 气泡），视觉上与对话内容明确区分，如"📋 你从菜单中选了以下菜品"

**优先级**：`[MVP]`

### F08 — 点餐单与展示模式

**用户目标**：整理好点的菜，展示给服务员。

**功能描述**：
- Chat 页面内浮动点餐单入口（有菜品时出现，显示数量 + 总价）
- Chat 右上角点单 badge 图标入口
- 点餐单页面：菜品列表 + 数量调整（+/-）+ 小计（减到 0 即删除，不单独设删除按钮）
- 菜名区域需处理长名称：单行截断 + ellipsis，避免双语长菜名撑破布局
- **展示给服务员模式（Waiter）**：全屏大字，只显示原文菜名 + 数量，**不显示价格**，屏幕常亮
- **指点式沟通面板（DEC-060）**：Waiter Mode 下点击任意菜品弹出双语沟通面板，提供预设短语（用户语言 + 餐厅当地语言双语显示），实现服务员与用户之间的跨语言沟通——SAGE 不只是点餐工具，也是餐桌上的沟通桥梁
- 货币符号自动适配（¥ / $ / € / ฿ 等）
- **Waiter 双出口**（DEC-040）：
  - 「继续点菜」→ 回到 Order（→ Chat → 继续加菜）
  - 「结束用餐」→ 清空会话 → 回到 Home

**导航规则**：
- Order ← 返回 → Chat
- Order → 展示给服务员 → Waiter
- Waiter → 继续点菜 → Order / 结束用餐 → Home

**验收标准**：
- AC1: 数量变化时总价实时更新
- AC2: 展示模式字号 ≥ 28px，高对比（深色背景白字）
- AC3: 展示模式只显示原文菜名 + 数量，不显示翻译、不显示价格
- AC4: 展示模式屏幕常亮（navigator.wakeLock，不可用时降级提示）
- AC5: 「继续点菜」回到 Order，可继续回 Chat 加菜
- AC6: 「结束用餐」需二次确认（"这将清空本次所有点菜记录，确定吗？"），确认后清空 Order + 会话 → Home（DEC-057）
- AC7: 进入 Waiter Mode 时，若用户有声明过敏原/饮食禁忌，顶部显示过敏/禁忌栏（图标 + 英文 + 本地语言三重表达），服务员第一眼可见（DEC-056）
- AC8: Waiter Mode 下点击任意菜品弹出指点式沟通面板（双语：🚫没有了 / 🔄换一道 / ➕加一份 / ❓其他），用于服务员-用户跨语言沟通（DEC-060）
- AC9: 进入 Waiter Mode 前，若 Order 含用户已知过敏原菜品，弹底部 sheet 提醒并要求确认（DEC-056）

**优先级**：`[MVP]`

> **小费建议**：MVP 不实现。美国市场推广时列为高优先级需求（见 Sprint 2）。

### F11 — 菜品概要（Dish Brief） `[Sprint 2]`

**用户目标**：面对不熟悉的菜名，快速了解"这道菜是什么、什么味道、适不适合我"。

**痛点**：用户在异国餐厅看到陌生菜名（如 Tom Kha Gai、Pad Kra Pao），仅凭名称和翻译无法判断菜品内容和口感。

**功能描述**：
- AI 识别菜单后，为**每道菜**生成概要介绍
- **默认展示**：一句话概要（食材组成 + 味道类型）
  - 示例："椰浆鸡汤，酸辣微甜，口感浓郁"
- **展开详情**：追加熟悉菜品类比 + 文化背景
  - 示例："类似酸辣汤但以椰奶打底。泰国国民级汤品，从街边到高档餐厅都有。"
- 概要在菜品卡片（F07 探索视图 + F06 对话内推荐卡片）中展示

**概要维度**：

| 维度 | 展示位置 | 说明 |
|------|---------|------|
| 食材组成 | 默认一句话 | 主要食材，3-5 个关键词 |
| 味道类型 | 默认一句话 | 酸/甜/辣/咸/鲜/浓郁/清淡等 |
| 熟悉菜品类比 | 展开详情 | "类似 XX 但 YY"，建立直觉认知 |
| 文化背景 | 展开详情 | 一句话，菜品在当地的地位/场景 |

**验收标准**：
- AC1: 每道菜都有一句话概要，无遗漏
- AC2: 默认折叠状态下只显示一句话，点击可展开
- AC3: 概要语言与用户界面语言一致（F10）
- AC4: 概要内容由 AI 在菜单识别阶段一并生成，不额外增加 API 调用
- AC5: 展开/折叠动画流畅，无卡顿

**优先级**：`[Sprint 2]`

### F12 — 饮食标签与过敏原标记（Dietary Tags） `[Sprint 2]`

**用户目标**：一眼识别菜品是否安全（过敏原）、是否符合饮食习惯（清真/素食/辣度/热量）。

**痛点**：国际用户对过敏原极度敏感，宗教/生活方式饮食限制是刚需，AI 翻译菜名无法解决这些信息缺失。

**功能描述**：

**标签体系**：

| 类别 | 标签 | 说明 |
|------|------|------|
| 过敏原 | 🥜花生、🦐甲壳类、🌾麸质、🥛乳制品、🥚蛋、🫘大豆、🌰坚果 | 8 大常见过敏原 |
| 宗教/生活方式 | ☪️清真、🥬素食、🌱纯素 | 核心 3 种 |
| 辣度 | 🌶×1~5 | 5 级辣度 |
| 营养 | ~XXX kcal | 卡路里估算，波浪号表示约值 |
| 其他 | 生食、含酒精 | 补充性标签 |

**标签来源策略**：
- **优先**：菜单上已标注的信息（如菜单标了 V = 素食）
- **补充**：AI 基于菜品信息推断
- **不确定时**：标注"⚠️ 可能含有 XX"，**宁可多标不漏标**

**与用户偏好（F09）联动**：
- **标签高亮**：用户在 F09 设了过敏原 → 匹配的标签变红底白字 + ⚠️ 图标
- **警告条**：匹配到用户过敏原时，菜品卡片顶部显示橙色警告条："⚠️ 可能含有您标记的过敏原：花生"

**验收标准**：
- AC1: 每道菜至少标注辣度 + 卡路里；过敏原和饮食标签按需标注
- AC2: 菜单已标注的信息优先于 AI 推断
- AC3: 不确定的过敏原标注"可能含有"，不静默跳过
- AC4: 匹配用户 F09 过敏原设置时，标签高亮 + 卡片顶部警告条
- AC5: 卡路里展示格式为"~350 kcal"
- AC6: 标签在菜品卡片（F07 + F06 推荐卡片）中紧凑展示，不占过多空间
- AC7: 标签数据由 AI 在菜单识别阶段一并生成，不额外增加 API 调用

**优先级**：`[Sprint 2]`

### F13 — 语音输入（Voice Input） `[Sprint 3]`

**用户目标**：在餐厅场景下用语音与 AI 对话，比打字更快更自然。

**痛点**：餐厅环境下打字不便（一手拿菜单、光线暗、赶时间），语音交流更符合真实场景。

**功能描述**：
- AgentChat 输入栏增加🎤按钮
- **按住说话**：按住录音，松手发送（转为文字后发给 AI）
- 支持**多语言**识别（用户可能用中文问英文菜单的内容）
- MVP 只做语音**输入**，不做语音输出（TTS）

**技术方向**：
- 优先：Web Speech API（浏览器原生，零成本）
- 降级：不支持时显示提示，隐藏🎤按钮

**验收标准**：
- AC1: 按住🎤录音，松手后自动转文字并发送
- AC2: 支持中文和英文语音识别
- AC3: 浏览器不支持 Web Speech API 时，🎤按钮隐藏，不影响文字输入
- AC4: 录音过程中有视觉反馈（如波形动画或脉冲效果）
- AC5: 转写结果发送前允许用户确认/编辑（可选，v1 可跳过直接发送）

**优先级**：`[Sprint 3]`

## 4.5 全局导航架构（DEC-040）

**导航模式**：纯任务流，无 Tab Bar。

```
Home ──→ Scanner ──→ Chat ←──→ Explore
  │         ↑          ↕           ↕
  ⚙        │        Order ←──→ Waiter
Settings    │          ↑           ├→ 继续点菜 → Order
            │          │           └→ 结束用餐 → Home（清空）
            │          │
            └──────────└── Explore 也可直达 Waiter / Chat（DEC-043）
```

**Chat 是中枢**：进入 Chat 后，所有子功能（Explore / Order / Waiter / 补拍）都从 Chat 出发、回到 Chat。

**Explore 双出口**（DEC-043）：用户在 Explore 选完菜后，可直接展示给服务员（→Waiter），或带已选菜品咨询 AI（→Chat）。

**导航状态机规则（DEC-057）**：
1. Home → Scanner → Chat 为单向主路径，Chat 返回 Home 需二次确认并清空会话
2. Chat ↔ Explore ↔ Order 三者可自由切换，状态互相保留
3. Order → Waiter 为单向进入，Waiter 无"返回 Order"（只有"继续点菜"重新进入 Order）
4. Waiter「结束用餐」→ 二次确认 → 清空全部状态 → Home
5. 任何页面的"清空"操作都需二次确认弹窗
6. Explore 有双出口：→ Waiter（直接展示）或 → Chat（咨询 AI）

**返回规则**：
| 页面 | ← 返回 | 特殊行为 |
|------|--------|---------|
| Scanner | Home（首次）/ Chat（补拍） | — |
| Chat | Home | 需二次确认，确认后清空会话 |
| Explore | Chat | — |
| Order | Chat | — |
| Waiter | — | 双出口：继续点菜→Order / 结束用餐→Home |
| Settings | Home | — |

**详细 Spec**：见 `docs/navigation-spec.md`

## 6. 非功能性需求

| 类别 | 要求 |
|------|------|
| **性能** | 首屏可交互 < 2s（4G）；Icebreaker < 1s；AI 推荐 < 5s（P90）|

| **安全** | AI API Key 仅在 Cloudflare Worker；前端零 Key |
| **隐私** | 菜单图片不持久存储；用户偏好仅存本地 localStorage |

| **网络中断恢复** | 网络中断时显示友好提示（非技术错误），已有对话和 Order 数据不丢失；网络恢复后自动重连并继续服务，无需用户手动刷新 |
| **国际化** | UI 中英双语；菜单识别支持 9 种核心语言 |
